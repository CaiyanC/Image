# 产品数据库字段清单（第一版）

> 基于《产品库元数据.xlsx》、`product_knowledge_base_plan.md`、以及两份视觉/元数据库方案整理。
> 目标：先落地基础产品库，再为后续 QA / RAG / 视觉素材管理预留扩展能力。

## 一、设计原则

- 以 `products` 作为主表，其他表统一通过 `product_id` 关联。
- 图片、视频、PDF 等文件不直接进数据库，数据库只存文件路径、文件名、类型、权限与审核状态。
- 第一版优先保证“能查商品、能查规格、能查卖点、能查内容、能查 QA、能查素材”。
- 字段名尽量与业务表/Excel 一致，避免后续迁移时歧义。

---

## 二、表结构总览

| 表名 | 作用 | 对应层级 |
|---|---|---|
| `products` | 产品主信息 | L1 |
| `product_specs` | 规格与物理信息 | L2 |
| `product_business` | 商业信息、卖点、场景 | L3 |
| `product_content` | 标题、文案、关键词 | L4 |
| `product_qa` | QA 知识库 | L5 |
| `product_media` | 图片/视频素材元数据 | L6 |
| `ai_generated_assets` | AI 生成资产与提示词 | L7 |

## 六、导入与使用建议

1. 先按这份字段清单建数据库。
2. 再把 `产品库元数据.xlsx`、`商品数据导出.xlsx` 做字段映射。
3. 图片/视频/文档文件存硬盘，数据库只记录路径与权限。
4. QA 后续可直接作为 RAG 知识源。
5. 视觉素材后续可继续补 `OCR 文本`、`caption`、`向量 ID` 等字段。

## 七、当前版本结论

这一版已经足够你先把 PostgreSQL 建起来，并导入第一批产品数据。后续如果字段还要扩展，建议尽量在不破坏主表结构的前提下增加子表或扩展字段。
