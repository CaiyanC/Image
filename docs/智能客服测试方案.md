# 智能客服回归测试方案

> 适用范围：每次修改智能客服、产品导入、QA 导入、向量检索、产品编辑/删除、前后端客服页面后，都按本文档测试一遍。

## 1. 测试目标

确认系统不是“规则客服”，而是能做到：

- 理解用户问题和历史对话，而不是只看关键词。
- 自主决定查询产品、QA、详情或生成待确认操作。
- 推荐时能结合人数、场景、容量、产品类型、目标人群做判断。
- 多轮对话能承接上下文，但不会被某个旧 SKU 锁死。
- 产品修改、删除、导入后，数据库和向量检索结果保持一致。
- 回答必须有溯源，能说明推荐依据和数据来源。

## 2. 测试前准备

### 2.1 启动系统

后端：

```powershell
cd backend
python -m uvicorn app.main:app --host 0.0.0.0 --port 8001 --reload
```

前端：

```powershell
cd frontend
npm run dev
```

浏览器打开前端页面，进入“智能客服”。

### 2.2 自动化基础检查

每次代码修改后先跑：

```powershell
$env:PYTHONPATH='backend'; python -m unittest backend.tests.test_customer_agent_service backend.tests.test_agent_action_service backend.tests.test_product_vector_index_service
```

前端修改后跑：

```powershell
cd frontend
npm run build
```

通过标准：

- 后端单元测试全部通过。
- 前端构建成功。
- 智能客服页面没有“当前 SKU”输入框。

## 4. 核心测试用例

### T01 三个年轻人露营推荐

用户问题：

```text
三个年轻人露营，适合带什么产品？
```

期望结果：

- 不要只罗列 10 多条数据库资料。
- 应该主动筛选 2-4 个适合产品。
- 应该优先考虑 3 人、露营、烹饪、容量、炉具搭配。
- 可以推荐如炊墨套锅、小青炉等组合，但必须解释为什么适合 3 人。
- 不应把 1000ML 单人锅作为主推荐；如果提到，只能说明它更适合单人或轻量徒步。
- 回答中应出现产品 SKU、名称、关键参数和推荐理由。

不通过表现：

- 输出“找到 14 条产品资料”这种原始列表。
- 不做判断，只复制数据库字段。
- 推荐明显不适合 3 人的产品且不解释限制。

### T02 泡咖啡小锅推荐

用户问题：

```text
适合泡咖啡的小锅有吗？
```

期望结果：

- 应优先推荐适合烧水、泡咖啡、容量较小、便携的锅具。
- 行山单锅 CW-C93 可以作为主推荐。
- 不应推荐炒锅、煎锅作为泡咖啡主方案。
- 如果提到大锅或煎锅，必须明确说“不优先推荐”或“不适合作为泡咖啡小锅”。

不通过表现：

- 把 3700ML 炒锅推荐给单人泡咖啡。
- 把煎盘当作泡咖啡小锅。

### T03 多轮上下文但不锁 SKU

按顺序提问：

```text
CW-C83 是什么？
```

```text
三个年轻人露营，适合带什么？
```

```text
那如果只是一个人徒步呢？
```

期望结果：

- 第二问不能因为第一问提过 CW-C83，就只推荐 CW-C83。
- 第二问应重新围绕“三个年轻人露营”判断候选产品。
- 第三问应理解“一个人徒步”是新条件，可以转向轻量、单人产品。
- 如果用户说“这个”，客服应结合最近一轮实际讨论对象判断；不确定时应追问澄清。

不通过表现：

- 后续所有问题都固定回答 CW-C83。
- 用户切换场景后仍强行推荐旧 SKU。

### T04 指代消解

按顺序提问：

```text
帮我推荐一个适合家庭露营的套锅
```

```text
这个容量是多少？
```

期望结果：

- 如果上一轮只推荐了一个明确产品，应回答该产品容量。
- 如果上一轮推荐了多个产品，应先说明“你指的是哪一个？”并列出候选 SKU。
- 不要凭空猜一个 SKU。

### T05 产品详情问答

用户问题：

```text
CW-C83-1 的容量和材质是什么？
```

期望结果：

- 应查询产品详情。
- 应准确回答容量和材质。
- 目前数据库中容量应类似“锅 3700ML”或已被测试修改后的最新值。
- 必须带 SKU 和产品名称。

### T06 修改容量并保留 label/value

操作前确认数据库中 CW-C83-1 容量结构，例如：

```json
[{"label":"锅","value":"3700ML"}]
```

用户问题：

```text
把 CW-C83-1 的容量改成 2000ml
```

期望结果：

- 系统生成“待确认修改”操作，而不是直接静默改库。
- 修改内容应保留 label/value 结构。
- 期望 draft/new value 类似：

```json
[{"label":"锅","value":"2000ml"}]
```

不通过表现：

- 把容量改成纯字符串，丢失 label。
- 出现“label 不知道怎么办”的异常。
- 没有生成待确认操作。

测试后处理：

- 如果只是测试，不要确认修改；或者确认后再改回原值。

### T07 删除产品

用户问题：

```text
删除 CW-C83
```

期望结果：

- 系统生成待确认删除操作。
- 不应出现 `Object of type UUID is not JSON serializable`。
- 回答应明确提示这是高风险操作，需要确认。
- 未确认前数据库产品不应被删除。

测试后处理：

- 不要确认删除真实产品。

### T08 QA 问答命中

准备条件：

- 系统中存在至少一条产品 QA，例如“这个锅可以用什么炉头？”

用户问题：

```text
这个锅可以用什么炉头？
```

期望结果：

- 如果上下文中产品明确，应结合该产品 QA 回答。
- 如果上下文中产品不明确，应追问是哪个 SKU。
- 回答应优先使用 QA 或产品资料，不要编造。

### T09 产品导入后检索

操作：

1. 导入一个新产品，名称、SKU、容量、场景、卖点都填写完整。
2. 导入完成后在产品列表确认数据库已出现该产品。
3. 去智能客服提问新产品相关问题。

用户问题示例：

```text
有没有适合双人徒步的新款轻量锅？
```

期望结果：

- 新导入产品能被客服检索到。
- 回答中能出现新产品 SKU 或名称。
- 如果新产品更匹配，应被推荐到前面。
- 溯源中能看到来自产品数据。

不通过表现：

- 产品列表有数据，但客服完全搜不到。
- 只能按精确 SKU 搜到，模糊场景搜不到。

### T10 QA 导入后检索

操作：

1. 给某个产品导入一条新 QA。
2. 用自然语言问同义问题，不要照抄 QA 原句。

用户问题示例：

```text
这个锅清洗麻烦吗？
```

期望结果：

- 客服能命中或接近命中新导入 QA。
- 回答应结合 QA 内容，不应只泛泛回答。
- 如果 QA 和产品字段都有相关信息，应综合说明。

### T11 修改产品后向量同步

操作：

1. 修改某产品卖点或场景。
2. 保存后提问修改后的关键词。

用户问题示例：

```text
有没有适合高海拔徒步、快速烧水的锅？
```

期望结果：

- 修改后的内容能影响客服检索和推荐。
- 旧内容不应继续主导回答。
- 如果系统有重建索引入口，修改后应确认索引已同步或重建完成。

### T12 无数据或不确定场景

用户问题：

```text
我想送礼，预算不确定，送给刚开始露营的朋友，有什么建议？
```

期望结果：

- 客服应先理解“送礼”“新手”“露营”。
- 可以给 2-3 个方向，并说明适合原因。
- 如果缺少预算、人数、交通方式，可以温和追问。
- 不应机械要求用户必须提供 SKU。

## 5. 回答质量评分标准

每个回答按 5 分制评分：

| 维度 | 1 分 | 3 分 | 5 分 |
| --- | --- | --- | --- |
| 意图理解 | 明显跑题 | 大致理解 | 准确理解隐含需求 |
| 产品匹配 | 只关键词命中 | 有一定筛选 | 能结合人数、场景、容量判断 |
| 上下文 | 忘记上下文或锁死旧 SKU | 偶尔能承接 | 能承接，也能识别场景切换 |
| 溯源 | 无依据 | 有 SKU/字段 | 有 SKU、字段、推荐逻辑 |
| 表达 | 数据堆砌 | 基本清楚 | 像专业客服，简洁可执行 |

通过标准：

- 核心用例 T01-T07 必须全部通过。
- 单个回答总分不低于 20 分。
- 不能出现严重错误：编造产品、误删数据、UUID 序列化报错、旧 SKU 锁死。

## 6. 发现问题时怎么记录

每个失败问题都记录：

- 用户完整问题。
- 前 3 轮历史对话。
- 客服完整回答。
- 是否有工具调用结果或溯源。
- 期望它怎么答。
- 是检索问题、推理问题、数据问题、前端展示问题，还是操作执行问题。

建议格式：

```text
问题编号：
用户问题：
历史上下文：
实际回答：
期望回答：
判断类型：检索 / 推理 / 数据 / 前端 / 操作
严重程度：高 / 中 / 低
```

## 8. 最终验收标准

可以认为本轮智能客服测试通过，当且仅当：

- 用户模糊提问时，客服能主动分析，而不是返回原始资料列表。
- 用户切换场景时，客服不会被旧 SKU 锁死。
- 用户要求修改/删除时，系统只生成待确认操作，不直接误操作。
- 产品和 QA 新增、修改后，客服检索结果能同步更新。
- 回答读起来像专业销售/客服，而不是数据库字段搬运工。
